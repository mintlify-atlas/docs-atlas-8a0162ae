## Overview

The Echo API uses **Supabase Auth** for authentication. All API requests must include a valid Bearer token in the `Authorization` header. Tokens are obtained through Supabase authentication and validated on each request.

<Note>
All API endpoints require authentication unless explicitly stated otherwise. Requests without valid authentication will receive a `401 Unauthorized` response.
</Note>

## Authentication Flow

Echo validates authentication tokens using the following process:

1. Extract the Bearer token from the `Authorization` header
2. Validate the token with Supabase Auth using `supabase.auth.getUser()`
3. Attach the authenticated user to the request context
4. Proceed with the API request if valid

## Bearer Token Format

All authenticated requests must include an `Authorization` header with the following format:

<ParamField header="Authorization" type="string" required>
  Bearer token in the format: `Bearer <your_access_token>`
</ParamField>

### Example Header

```http
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
```

## Making Authenticated Requests

### cURL Example

```bash
curl -X GET https://api.echo.app/api/profiles/user_123 \
  -H "Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..." \
  -H "Content-Type: application/json"
```

### JavaScript Example

```javascript
const accessToken = 'your_supabase_access_token';

const response = await fetch('https://api.echo.app/api/profiles/user_123', {
  method: 'GET',
  headers: {
    'Authorization': `Bearer ${accessToken}`,
    'Content-Type': 'application/json'
  }
});

const data = await response.json();
```

### Python Example

```python
import requests

access_token = 'your_supabase_access_token'

headers = {
    'Authorization': f'Bearer {access_token}',
    'Content-Type': 'application/json'
}

response = requests.get(
    'https://api.echo.app/api/profiles/user_123',
    headers=headers
)

data = response.json()
```

## Obtaining Access Tokens

Access tokens are obtained through Supabase authentication. You can get tokens by:

1. **User Login**: Authenticate users via Supabase Auth (email/password, OAuth, magic links)
2. **Session Management**: Retrieve the session token from the authenticated Supabase client
3. **Token Refresh**: Use refresh tokens to obtain new access tokens when they expire

### Supabase Client Example

```javascript
import { createClient } from '@supabase/supabase-js';

const supabase = createClient(
  'your_supabase_url',
  'your_supabase_anon_key'
);

// Sign in a user
const { data, error } = await supabase.auth.signInWithPassword({
  email: 'user@example.com',
  password: 'password123'
});

// Get the access token
const accessToken = data.session?.access_token;
```

## Error Responses

### Missing Authorization Header

If the `Authorization` header is missing or doesn't start with "Bearer ":

```json
{
  "error": "Unauthorized",
  "message": "Missing or invalid authorization header"
}
```

**Status Code**: `401 Unauthorized`

### Invalid or Expired Token

If the token is invalid, expired, or cannot be verified:

```json
{
  "error": "Unauthorized",
  "message": "Invalid or expired token"
}
```

**Status Code**: `401 Unauthorized`

### Authentication Service Error

If there's an internal error during authentication:

```json
{
  "error": "Internal Server Error",
  "message": "Authentication failed"
}
```

**Status Code**: `500 Internal Server Error`

## Security Best Practices

<Warning>
Never expose your access tokens in client-side code, version control, or logs. Always store tokens securely and transmit them over HTTPS.
</Warning>

- **Use HTTPS**: Always make API requests over HTTPS to prevent token interception
- **Token Storage**: Store tokens securely (e.g., HTTPOnly cookies, secure storage)
- **Token Expiration**: Implement token refresh logic to handle expired tokens
- **Scope Limitations**: Users can only access resources they own (enforced by user ID validation)
- **Environment Variables**: Store Supabase credentials in environment variables, never in code

## User Context

Once authenticated, the API attaches user information to the request context:

- `user`: Full Supabase user object
- `userId`: The authenticated user's unique ID

Many endpoints verify that the requested resource belongs to the authenticated user before allowing access. For example, users can only access their own profile data.
